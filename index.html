Fastjson反序列化分析

简介

Fastjson 是阿里巴巴的开源 JSON 解析库，它可以解析 JSON 格式的字符串，支持将 Java Bean 序列化为 JSON 字符串，也可以从 JSON 字符串反序列化到 JavaBean，Fastjson不但性能好而且API非常简单易用，所以用户基数巨大，一旦爆出漏洞其影响对于使用了Fastjson的Web应用来说是毁灭性的。

Fastjson 使用

将类转为 json

在这里我们最常用的方法就是 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">JSON.toJSONString()</font>

将 json 反序列化为类

将 json 数据反序列化时常使用的方法为<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">parse()</font>、<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">parseObject()</font>、<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">parseArray()</font>

特性

  • 使用 <font style="background-color:rgb(247, 247, 247);">JSON.parse(jsonString)</font> 和 <font style="background-color:rgb(247, 247, 247);">JSON.parseObject(jsonString, Target.class)</font>，两者调用链一致，前者会在 jsonString 中解析字符串获取 <font style="background-color:rgb(247, 247, 247);">@type</font> 指定的类，后者则会直接使用参数中的class。
  • fastjson 在创建一个类实例时会通过反射调用类中符合条件的 getter/setter 方法，其中 getter 方法需满足条件：方法名长于 4、不是静态方法、以 <font style="background-color:rgb(247, 247, 247);">get</font> 开头且第4位是大写字母、方法不能有参数传入、继承自 <font style="background-color:rgb(247, 247, 247);">Collection|Map|AtomicBoolean|AtomicInteger|AtomicLong</font>、此属性没有 setter 方法；setter 方法需满足条件：方法名长于 4，以 <font style="background-color:rgb(247, 247, 247);">set</font> 开头且第4位是大写字母、非静态方法、返回类型为 void 或当前类、参数个数为 1 个。具体逻辑在 <font style="background-color:rgb(247, 247, 247);">com.alibaba.fastjson.util.JavaBeanInfo.build()</font> 中。
  • 使用 <font style="background-color:rgb(247, 247, 247);">JSON.parseObject(jsonString)</font> 将会返回 JSONObject 对象，且类中的所有 getter 与setter 都被调用。
  • 如果目标类中私有变量没有 setter 方法，但是在反序列化时仍想给这个变量赋值，则需要使用 <font style="background-color:rgb(247, 247, 247);">Feature.SupportNonPublicField</font> 参数。
  • fastjson 在为类属性寻找 get/set 方法时，调用函数 <font style="background-color:rgb(247, 247, 247);">com.alibaba.fastjson.parser.deserializer.JavaBeanDeserializer#smartMatch()</font> 方法，会忽略 <font style="background-color:rgb(247, 247, 247);">_|-</font> 字符串，也就是说哪怕你的字段名叫 <font style="background-color:rgb(247, 247, 247);">_a_g_e_</font>，getter 方法为 <font style="background-color:rgb(247, 247, 247);">getAge()</font>，fastjson 也可以找得到，在 1.2.36 版本及后续版本还可以支持同时使用 <font style="background-color:rgb(247, 247, 247);">_</font> 和 <font style="background-color:rgb(247, 247, 247);">-</font> 进行组合混淆。
  • fastjson 在反序列化时，如果 Field 类型为 <font style="background-color:rgb(247, 247, 247);">byte[]</font>，将会调用<font style="background-color:rgb(247, 247, 247);">com.alibaba.fastjson.parser.JSONScanner#bytesValue</font> 进行 base64 解码，对应的，在序列化时也会进行 base64 编码。

如何才能触发反序列化得到的恶意类中的恶意函数呢？

Fastjson 反序列化漏洞的核心在于其 autoType 功能。当 autoType 开启时，Fastjson 会根据 JSON 数据中的 @type 字段来实例化对应的 Java 对象。

由前面知道，在某些情况下进行反序列化时会将反序列化得到的类的构造函数、getter方法、setter方法执行一遍，如果这三种方法中存在危险操作，则可能导致反序列化漏洞的存在。

RMI机制分析

什么是RMI

Java RMI（Remote Method Invocation，远程方法调用）是一种用于在分布式系统中进行远程通信的机制。它允许一个Java虚拟机（JVM）上的对象调用另一个JVM上的对象方法，就像调用本地方法一样。

RMI 的核心概念

  1. 远程对象：存在于远程JVM中的对象，客户端可以通过网络调用其方法。
  2. 存根（Stub）和骨架（Skeleton）：
    • Stub：客户端的代理，负责将调用请求发送到远程对象。
    • Skeleton：服务器端的代理，负责接收请求并调用实际对象的方法。
  3. RMI Registry：一个简单的命名服务，客户端通过它查找远程对象的引用。

RMI 的工作流程

  1. 定义远程接口：接口必须继承 <font style="color:rgb(64, 64, 64);">java.rmi.Remote</font>，方法需声明抛出 <font style="color:rgb(64, 64, 64);">RemoteException</font>。
  2. 实现远程接口：实现类通常继承 <font style="color:rgb(64, 64, 64);">UnicastRemoteObject</font>。
  3. 启动RMI Registry：使用 <font style="color:rgb(64, 64, 64);">rmiregistry</font> 命令启动注册表。
  4. 注册远程对象：将远程对象绑定到RMI Registry。
  5. 客户端查找远程对象：通过RMI Registry获取远程对象的引用并调用方法。

RMI 的用途

  1. 分布式计算：将任务分布到多台机器上执行。
  2. 远程服务调用：客户端调用远程服务器上的服务。
  3. 负载均衡：通过远程调用将请求分发到不同服务器。
  4. 模块化设计：将系统拆分为多个独立模块，部署在不同服务器上。

内存马分析

前言

内存马是一种将恶意代码直接注入到应用程序运行时内存中的技术手段。与传统的基于文件的WebShell不同，内存马不依赖于磁盘上的任何持久化存储，而是利用Java、.NET等语言提供的反射机制或动态编译功能，在内存中创建并执行恶意逻辑。这种方式不仅能够绕过大多数基于文件扫描的安全防护措施，而且由于其高度的灵活性和隐蔽性，使得一旦成功植入，很难被发现和清除。

java内存马常见类型

  • 动态注册servlet/filter/listener(使用servlet-api的具体实现）
  • 动态注册interceptor/controller(使用框架如spring/struts2)
  • 动态注册使用职责链设计模式的中间件、框架的实现（例如Tomcat的Pipeline&Valve，Grizzly 的 FilterChain & Filter等等）
  • 使用java agent技术写入字节马

Servlet-API类

Servlet-API 是一套标准规范（JSR），其核心组件包括 Servlet、Filter、Listener 三大接口。在启动时，Tomcat（作为 Servlet 容器）通过扫描 web.xml、解析注解（如 @WebServlet）、以及处理编程式动态注册的方式来发现这些组件的配置信息。然后，Tomcat 的类加载器会加载这些接口的实现类，并通过反射机制实例化它们，最终将其纳入相应的生命周期中进行管理。

Servlet 处理请求，Filter 拦截请求，Listener 监听事件；三者共同构成 Java Web 的生命周期与请求处理链。

可以类比工厂，Servlet-API是图纸，容器是工厂，Servlet、Filter、Listener是根据图纸造出来的工人， 内存马做的事情就是——半夜溜进工厂，直接往花名册里添名字，让工厂第二天“凭空”多出一个没人见过的员工。

servlet内存马

Servlet内存马是通过动态注册Servlet来实现的一种内存攻击手段。在JavaWeb应用中，Servlet作为处理客户端请求的核心组件之一，能够直接处理HTTP请求并返回响应。攻击者利用这一点，通过程序化地向Web容器（如Tomcat）在运行时注器恶意的Servlet对象，使得该Servlet能够在没有实际文件存在的情况下执行恶意程序。

filter内存马

Filter内存马是通过动态注册Filter来实现的一种内存攻击手段。在JavaWeb应用中，Filter作为拦截HTTP请求与响应的核心组件之一，能够对进入Web容器的所有请求进行预处理和过滤。攻击者利用这一点，通过程序化地向Web容器（如Tomcat）在运行时注入一个恶意的Filter对象，使得该Filter能够在没有实际文件存在的情况下，以最高优先级拦截并处理所有请求，从而执行恶意代码。

相较于Servlet，Filter的拦截优先级更高，且不局限于特定路径（一个映射为 /* 的Filter可以拦截到应用中的所有请求），隐蔽性更强，拦截范围更广

Tomcat Valve内存马

Tomcat Valve（阀门）是一种拦截器或过滤器机制，它允许你在请求到达目标Servlet之前和响应返回给客户端之后，插入自定义的处理逻辑。与Filter不同的是，Valve的作用域为tomcat容器，而Filter的作用域为web应用。因此，Valve更底层，可以影响所有部署的应用，无需修改应用代码。

到了这里不得不扯一下 Tomcat 中的Pipeline-Valve 的管道机制 ，当我们向一个网站传入http请求时，我们的请求会经过这四个部分：

  1. Engine（引擎）：Tomcat 的顶级容器，代表整个 Catalina Servlet 引擎。
  2. Host（虚拟主机）：代表一个虚拟主机，通常对应一个域名。它实现了基于域名的托管，允许你在同一台服务器上运行多个不同域名的网站。
  3. Context（上下文）：代表一个独立的 Web 应用程序。
  4. ** Wrapper（包装器）：**是最底层的容器，每个 Wrapper 封装和管理一个特定的 Servlet。

Tomcat 的每个容器（Engine、Host、Context、Wrapper）都有自己的一个 Pipeline（管道）对象，用来管理处理流程。Valve（阀门）就是你在 server.xml 或 context.xml 中配置的 **<Valve>** 组件。每个 Valve 都负责一项特定的前置任务，比如记录日志、检查权限、过滤IP等。每个 Pipeline 必须且只有一个 Basic Valve（基础阀门），它负责最核心、最最终极的任务。

我们在Servlet的doGet方法处下断点可以发现，当一个http请求到达Servlet时会经过大大小小很多Valve，也可以验证上述说法

Agent基础

是什么？

Java Agent 是 Java 平台提供的一种极其强大的“元编程”工具，它允许开发者在 JVM 层面操控和观测应用程序，是实现许多高级开发、运维工具的技术基石。

与普通Java程序通过main方法启动不同，Java Agent并不是一个可以单独启动的程序，它必须依附在一个Java应用程序（JVM）上，与主程序运行在同一个JVM进程中，通过Instrumentation 接口与JVM进行交互。

Java Agent提供了一种在加载字节码时对字节码进行修改的能力，有两种执行方式：

一、在应用运行之前，通过premain()方法来实现「在应用启动时侵入并代理应用」，这种方式是利用Instrumentation接口实现的；

二、在应用运行之后，通过Attach API和agentmain()方法来实现「在应用启动后的某一个运行阶段中侵入并代理应用」，这种方式是利用Attach接口实现的。

内存马

核心原理

Agent 注入内存马的核心原理是利用 Java Agent 的 Instrumentation 机制，在运行时动态修改已加载到 JVM 中的类的字节码，将恶意逻辑（如 Webshell 的处理逻辑）“编织”到某个特定的、会被频繁调用的类和方法中（例如 Filter、Servlet、Controller 等）。

什么是Instrumentation

  1. Instrumentation 是 JVMTIAgent（JVM Tool Interface Agent）的一部分，Java agent通过这个类和目标 JVM 进行交互，从而达到修改数据的效果
  2. 在 Instrumentation 中增加了名叫 transformer 的 Class 文件转换器，转换器可以改变二进制流的数据
  3. Transformer 可以对未加载的类进行拦截，同时可对已加载的类进行重新拦截，所以根据这个特性我们能够实现动态修改字节码

在Instrumentation类中，以下几个办法比较重要

这几个方法涉及到了Transformer的调用机制，当我们获取到了Instrumentation，我们要怎么插入恶意代码到目标主机呢？答案是将恶意逻辑写入transform方法，并利用机制触发其方法

transform方法的调用时机有两个：

  • 类加载时：当类被加载时，如果已经注册了Transformer，则会在类定义之前调用transform方法。
  • 重转换时：当对已加载的类进行重转换（通过retransformClasses）时，会调用transform方法。

当调用 Instrumentation.retransformClasses(Class<?>... classes) 时，JVM 会重新处理指定类的字节码。具体步骤包括：

  • 对于每一个要重转换的类，JVM 会调用所有注册的 ClassFileTransformer 的 transform 方法（这些转换器在注册时设置了 canRetransform 为 true）。
  • 每个转换器都有机会修改类的字节码。
  • 最后，JVM 使用修改后的字节码重新定义这个类。

具体到Instrumentation的使用，攻击者会通过以下步骤：

  • 获取到Instrumentation实例（通过agentmain或premain）。
  • 定义一个ClassFileTransformer，在transform方法中，针对目标类（如org.apache.catalina.core.ApplicationFilterChain）进行字节码修改，插入恶意逻辑。
  • 使用Instrumentation的addTransformer方法注册这个Transformer，并设置canRetransform为true。
  • 对于已经加载的类，调用retransformClasses方法触发重新转换，从而激活Transformer。

VirtualMachine

VirtualMachine 可以来实现获取系统信息，内存dump、现成dump、类信息统计（例如JVM加载的类）。通过 VirtualMachine 类的 attach(pid) 方法，可以 attach 到一个运行中的 java 进程上，之后便可以通过 loadAgent(agentJarPath) 来将agent 的 jar 包注入到对应的进程，然后对应的进程会调用agentmain方法。
