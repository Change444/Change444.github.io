域委派攻击

前言

域委派是一种机制，它允许某个计算机或服务账号在已通过身份验证的用户的上下文中，代表该用户去访问网络中其他计算机或服务上的资源。

例如在下图中jack要通过web服务器来访问后台文服，但是服务账号websvc本身是没有权限访问文服的，此时，就要websvc就要代表jack去访问文服，这就是委派。

委派的目的是为了解决多层级服务的访问问题，让用户的凭据不会在每一层都传递，而是通过委派机制让后端服务相信前端服务是代表合法用户来访问的。

委派的分类

非约束委派

这是最早的实现方式，极不安全。它的工作原理是：

  1. jack访问 WEB 服务的时候将以 Kerberos 协议向 KDC 申请访问 websvc 的 ST 票据。KDC 发现 websvc 配置了非约束性委派，于是在返回的 TGS 响应中，额外附带了一个可转发的 jack 的 TGT
  2. jack 将收到的 TGT 和 ST 一起发送给 websvc，websvc将jack的TGT缓存在内存中
  3. 当 websvc 需要代表 jack 访问后台文件服务器时，它直接使用内存中 jack 的 TGT，向 KDC 申请访问文件服务器的 ST 票据。
  4. KDC 验证 jack 的 TGT 有效后，检查 jack 是否有权访问文件服务器。如果有，则返回针对文件服务器的 ST 票据（代表 jack 的身份）。
  5. websvc 使用该 ST 票据向文件服务器发起请求，文件服务器验证票据后认为操作者是 jack，并根据 jack 的权限返回数据。

在上述过程中websvc会存储jack的TGT,这就导致websvc可以无限制地模拟用户A去访问网络上的任何服务。如果一个配置了非约束性委派的服务器（如Web服务器）被攻陷，攻击者就可以在服务器内存中抓取所有访问过该服务器的用户的TGT，从而轻松接管这些用户的身份，包括域管理员。

特点

  1. 非约束性委派的目标的 ACL 属性中会包：TRUSTED_FOR_DELEGATION。
  2. 委派的设置需要 SeEnableDelegation 特权，通常仅域管理员有。
  3. 域控主机默认是非约束性委派。

约束性委派

由于非约束委派存在过度授权、风险过高的问题。微软在 Windows Server 2003 中引入了约束委派。

在了解它的原理之前先要了解S4U2Self 与 S4U2Proxy这两个协议

  1. S4U2Self (Service for User to Self) ：允许服务为自己获取一个代表指定用户的可转发服务票据（Forwardable Service Ticket），即使该用户尚未通过 Kerberos 向此服务认证过。
  2. S4U2Proxy (Service for User to Proxy) ：允许服务将 S4U2Self 获取的用户票据（或用户直接提供的票据）转发给 KDC，以请求访问后端服务的票据。

这两个协议是配合使用的，一般流程是：

  1. 用户通过 Kerberos 向KDC申请访问前端服务（websrv​）的服务票据（ST），验证后KDC会颁发一个针对websrv的服务票据（ST1）

  2. 用户拿着这个ST1向websrv建立安全会话，确认用户身份

  3. 当前端服务需要访问后端服务时，websrv​会使用自己的机器账号（WEBSRV$​）的 TGT，向 KDC 发起一个 S4U2Proxy 请求。

  4. KDC 收到请求后，进行两项检查：

    • 验证 WEBSRV$ 是否具有约束委派权限。
    • 检查 WEBSRV$​ 的 msDS-AllowedToDelegateTo 属性中是否包含该服务（也就是白名单）

  5. 检查通过后，KDC会生成一个针对指定后端服务的服务票据（ST2）并返回

  6. 前端服务拿着这个票据访问后端服务，确认当前操作者的身份为用户

这样就避免了用户的TGT留在服务账号上，相对非约束委派安全性得到了提升，这里用ruyue师傅的图来总结一下流程

用户通过非Kerberos协议（如NTLM、表单登录）访问前端服务是无法获取票据ST1的，这时候就要通过S4U2Self协议，让服务主机代替向KDC申请服务票据，这里再放一张ruyue师傅的图方便理解

存在的问题

S4U2Self 请求的本质是向KDC发起一个S4U2Self请求，KDC在接受这个请求之后只会做三个校验：

  1. 验证请求是否真的是由websrv$发起的，而不是被伪造的
  2. 检查WEBSRV$的账户属性，看它是否被授予了协议转换（Protocol Transition） 的权限
  3. 验证目标是否允许被委派

KDC并不会管用户是否同意被委派，除非该用户被设置为敏感账户，这就导致如果如果一台被允许约束性委派的机器被控制后，就可以利用这个服务账号代表任意用户进行 S4U2self 获得 ST1，然后把获取到的票据用于 S42proxy，从而获取另一个服务的 ST2
