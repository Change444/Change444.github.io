Java代码审计基础

前言

本篇主要是对代码审计学习历程的记录和总结，包括一些常见漏洞的审计技巧、该如何开始代码审计以及代码审计需要的一些基础知识

基础知识

需要掌握的一些知识点

为了看懂代码，首先要掌握一些基本的开发思想，了解项目开发的结构，以下是一些常见的框架：

  1. JavaWeb（掌握）：动态⽹⻚开发技术Servlet&Jsp；服务容器Tomcat使⽤ & HTTP协议。

  2. Java开发框架（掌握）：Web开发框架SpringMVC、Struts2；持久层开发框架Mybatis、Hibernate、服务快速开发技术SpringBoot；项⽬构建Maven。

  3. 中间件&服务框架（了解）：微服务注册中⼼Spring Cloud Eureka；微服务调⽤Spring Cloud OpenFeign；微服务⽹关Spring Cloud Gateway；微服务统⼀配置中⼼Spring Cloud Alibaba Nacos；客户端负载均衡Spring Cloud Ribbon；微服务部署Docker。

项目包的命名规范

Java的包名都有⼩写单词组成，类名⾸字⺟⼤写；包的路径符合所开发的系统模块的定义，了解包的命名规范能帮助我们更快找到对应模块。常见的命名规范为：com.公司名.项⽬名.模块名…….

重点看模块名：

控制器包名（web接⼝相关）：controller、servlet、action、web

持久层包名（操作数据库相关）：dao、persist、mapper

实体类包名：entity、model、bean、javabean、pojo

过滤器：filter

监听器：listener

⼯具类：util

全局配置 ：global

语⾔环境、框架等配置⽂件：config

在不同的框架下⼀般包的命名规则不同，但⼤概如上，不同功能的 Java ⽂件放在不同的包中，根据Java ⽂件的功能统⼀安放及命名。

审计工具及环境配置

反编译源码

一般我们代码审计拿到的源码都是jar文件或者class文件，需要将其进行反编译。一个个找太麻烦了，我的办法是写一个脚本先筛选出业务相关的jar包，然后通过反编译工具将jar反编译为源码，最后再导入IDEA

原理判断jar中是否有指定目录结构，例如帆软com/fr/ 或 com/fanruan/，如果存在则将这个jar导出

处理完后会获得一个全是业务jar包的目录

将其打包，然后通过jadx-gui打开，保存并反编译为源代码即可批量获取项目源码了

之后，在idea中添加为代码源即可全局搜索和分析了

远程调试

在项目启动bat目录中设置中开启 JPDA 调试（默认监听8000端口）

然后在idea项目中配置jvm远程调试

将jar包导入项目，然后设置断点就能就远程调试了

SQL注入

SQLI(SQL Injection)， SQL注⼊是因为程序未能正确对⽤户的输⼊进⾏检查，将⽤户的输⼊以拼接的⽅式带⼊SQL语句，导致了SQL注⼊的产⽣。攻击者可通过SQL注⼊直接获取数据库信息，造成信息泄漏。

⽬前开发中进⾏数据库查询的⽅式有两种，⼀种是直接使⽤JDBC，写原⽣的SQL语句；另外⼀种使⽤ORM框架，减少编写SQL语句，例如Mybatis、Hibernate等。

JDBC

原生JDBC有两个⽅法执⾏SQL语句，分别是PrepareStatement和Statement。在Spring 框架中一般通过JdbcTemplate将JDBC查询封装

Statement

原始的Statement拼接语句，参数id为前端传⼊的，没有进⾏任何的过滤，直接拼接到SQL语句中，导致SQL注入

PrepareStatement

PrepareStatement会对SQL语句进⾏预编译，通过占位符的方式插入用户要查询的数据，保持查询语句和数据相分离，能从本质上避免SQL注⼊⻛险。

但是如果开发者为了便利，采用直接拼接的方式构造SQL，那预编译将失效

Mybatis

MyBatis框架底层已经实现了对SQL注⼊的防御，但存在使⽤不当的情况下，仍然存在SQL注⼊的⻛险。

主要的原因还是错误地使用${}

ORDER BY / GROUP BY / HAVING 子句

出现频率最高的点，主要原因是因为SQL关键字、库名、表名、字段名、函数名等字段不能被参数化（加引号），于是就不得不用${}

解决办法：排序映射，缺失是代码量会增加。如果想减少代码量，需要在Java层⾯做映射，设置⼀个字段/表名数组，仅允许⽤户传⼊索引值，这样保证传⼊的字段或者表名都在⽩名单⾥⾯。

XSS

XSS 是通过对⽹⻚插⼊可执⾏代码且成功地被浏览器执⾏，以达到攻击的⽬的。根据服务端的后端代码不同，XSS 的种类也不相同，⼀般可以分为反射型、存储型以及DOM 型。

反射性XSS

从/reflect接口获取input输入不做任何处理直接输出在页面上

存储型XSS

发布评论后被调用/stored接口将评论存储在comment中，前端调用comment时不经过处理将导致存储XSS

修复方案

  1. 基于后端防御，使⽤配置项⽬的全局过滤器，将输⼊的特殊字符转义后存储到数据库，或者过滤掉特殊字符，只保留正常数据；

  2. 基于前端防御，使⽤CSP、HttpOnly、前端特殊字符转义等⼿段进⾏防御。

全局搜索数据库的插入语句(关键词：insert,save,update)，然后找到该插入语句所属的方法名如(insertUser())，然后全局搜索该方法在哪里被调用，一层层的跟踪。直到找到getParamter()方法获取请求参数的地方停止，如果没有全局 XSS 过滤器，跟踪的整个流程都没有对获取的参数过滤，则存在存储型 XSS。

SSRF

SSRF（Server-Side Request Forgery，服务器端请求伪造） 是指攻击者利用服务端功能（如获取远程图片、调用API、爬取网页等）发起恶意请求，访问内网资源、本地文件或云元数据服务。

漏洞原因

主要还是由于未对输入url做校验导致，配合一些协议来对内网进行攻击，在java中，⼀般利⽤http/https协议来探测端⼝、暴⼒穷举等，还可以利⽤file协议读取/下载任意⽂件等

Java 原生 URL 类支持的协议

协议 支持情况 典型用途
​http​/https 完全支持 访问 Web 服务、云元数据、内网 HTTP 服务
​file 支持 读取本地文件（受文件权限限制）
​jar 支持 解压 JAR 文件中的条目，可配合file:​或http:
​ftp 支持（需显式启用） 访问 FTP 服务器（默认可能被禁用或需要 handler）
​mailto 不支持直接请求 仅解析，无法发起网络请求
​netdoc 支持 访问类路径资源（如netdoc:/foo/bar，用于 Java 文档）
​gopher 不支持 标准 JDK 无内置 gopher handler
​dict 不支持 无内置 handler

注意gopher和dict在java原生URL类中是不支持的，这点要跟php中的SSRF区分开来

修复方案

  1. 设置协议白名单，仅允许HTTP/HTTPS
  2. 每次请求前对域名进行解析，对ip进行安全检查，阻止访问内网、回环、链路本地等地址
  3. 禁用自动重定向，对重定向的地址进行递归检查，防止DNS重绑定

审计技巧​

  1. 列出所有出站请求代码块（HTTP、FTP、Socket、文件读取等）。

  2. 追踪每个请求 URL 的来源：是否来自用户输入（直接/间接）。

  3. 检查是否有校验逻辑：

    • 协议是否限制仅 HTTP/HTTPS？
    • 主机是否经过 DNS 解析并检查 IP 属于公网？
    • 端口是否白名单？
    • 是否处理了重定向？

  4. 尝试构造绕过：域名白名单绕过、IP 进制转换、重定向、DNS 重绑定等。

  5. 判断是否有回显：请求结果是否返回给用户（如显示图片、API 响应内容） → 回显 SSRF，危害更大。

  6. 检查网络边界：即使代码安全，内网 DNS 解析或云元数据地址未在网络层拦截也可能被利用。

反序列化

Java 反序列化是将字节流还原为 Java 对象的过程，反序列化过程中会创建对象，调用某些方法（如readObject、readResolve等），攻击者可以控制字节流中的类名、字段值，从而控制对象的状态甚至执行任意代码。

ObjectInputStream

Java原生反序列化，当反序列化的时候会自动调用readObject()​,readResolve()​(替换反序列化返回的对象),readObjectNoData()(处理类版本不匹配时的异常情况),并返回序列化对象。许大量 Java 组件（JMX、RMI、JNDI、WebLogic、JBoss）默认使用它。

漏洞点：readObject

FastJson

将 JSON 字符串还原为对象，会自动调用setter/getter

漏洞点：parseObject

SnakeYAML

将 YAML 字符串还原为对象，会自动调用无参构造方法和非公有属性的 setter 方法

漏洞点：load

XML反序列化

XML 反序列化是指将 XML 格式的文本（或输入流）还原为 Java 对象的过程，这段过程设计到类加载，有可能导致反序列化漏洞的出现，常见的有XMLDecoder和XStream。

XMLDecoder

XMLDecoder在JDK 1.4~JDK 11中都存在反序列化漏洞安全⻛险，XMLDecoder​ 将 XML 元素映射为 Java 对象创建和 setter 调用，并支持 <object>​、<void>​、<array>​ 等标签。其中 <void method="..."​ 可调用任意方法，<class> 可加载任意类。

XStream

XStream 默认允许指定类的全限定名，攻击者可以替换为恶意 gadget 类（如 java.lang.ProcessBuilder​）。即使业务期望一个普通 User​ 对象，攻击者仍可通过 <user class="com.evil.Payload"/> 注入危险类型。
