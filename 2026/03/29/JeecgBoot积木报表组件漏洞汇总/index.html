JeecgBoot积木报表组件漏洞汇总

queryFieldBySql接口freemark模版注入

漏洞版本

JeecgBoot <= 3.5.5

JimuReport <= 1.6.1

漏洞概述

由freemaker表达式解析导致的模板注入

本次分析的是3.5.0版本的jeecgboot，漏洞是积木报表这个组件导致的，由于JeecgBoot 集成了JimuReport ，所以JeecgBoot也存在这个漏洞

漏洞位于jimureport-spring-boot-starter-1.5.6.jar这个jar包中

漏洞分析

鉴权

jeecgboot是基于shiro+JWT来鉴权的

其中在ShiroConfig文件中定义了一系列免认证路由，访问这些路由无需身份校验，可被任意访问

而漏洞点正位于jmreport路由下，因此，触发这个漏洞不需要身份认证，在前台就可以访问这个路由

代码分析

路由/jeecg-boot/jmreport/queryFieldBySql​传入一段json数据{"sql":"select '<#assign ex="freemarker.template.utility.Execute"?new()> ${ ex("calc ") }' "}

下断点，var2会从传入的json中读出键为sql的数据然后调用i.a对sql语句进行判断，这里应该是做黑名单校验

往下走，这里执行了parseReportSql，这个方法接收4个参数，下面是对应在json中的键，漏洞利用只要传入sql即可

继续往下走，这里由于paramArray为空所以会走到else语句

继续，关注sql会怎么被处理

调用FreeMarkerUtils，这是一个工具类

这段代码是一个基于 FreeMarker 模板引擎的渲染工具方法。它的主要作用是将传入的模板字符串（var0​）与数据模型（var1）进行合并，生成最终的动态文本。

Freemarker调用点

​new Template("template", new StringReader(var0), var2)).process(var1, var3)会对freemarker表达式进行解析，而var1是我们在请求中传入的

跟进process方法，它设置当前线程的环境上下文，执行自动导入/包含doAutoImportsAndIncludes，然后从模板的根节点开始递归渲染

​visit(getTemplate().getRootTreeNode())

visit负责递归访问模板中的每一个 元素，逐元素执行具体的渲染逻辑

​Expression.eval​执行命令，这里会反射实例化我们指定的对象freemarker.template.utility.Execute，然后继续递归

随着他visit调用自身，来到了accept方法中，最终会走到Runtime.getRuntime().exec(aExecute);

小tips

值得注意的是，jeecgboot自己的Freemarker配置是安全的，但是JimuReport组件内部使用的是自己的Configuration实例 ，没有使用这个安全配置。因此导致了模版注入

loadTableData接口freemark模版注入

漏洞版本

JeecgBoot <= 3.5.5

JimuReport <= 1.6.1

漏洞概述

漏洞点和漏洞原理跟queryFieldBySql这个洞都差不多，就不过多赘述了

漏洞分析

代码位于jimureport-spring-boot-starter-1.5.6.jar的org.jeecg.modules.jmreport.desreport.a.a

会从json数据中读取：

  1. abSource
  2. tablename
  3. pageNo
  4. pageSize
  5. sql
  6. paramArray

读取sql传入var6，调用i.a对sql语句进行黑名单校验，这点上面讲过就不贴图了

首先会经过两个if，会对dbSource​进行判断，由于this.jmBaseConfig.getSaas()​返回的是false，所以将dbSource置为null即可绕过判断

接着进入i.loadTableData​,这里根据paramArray是否为空来执行不同的e.a

进入e.a后会执行freemaker调用，后面就跟前一个洞差不多了

是不是以为到这里就结束了？其实不行，直接打会提示sql报错

原因是freemaker表达式解析完后会将解析结构当作sql语句来查询，因此，还需传入pageSize和pageNo才能正常执行语句

upload接口任意文件上传

漏洞版本

JeecgBoot <= 3.5.5

JimuReport <= 1.6.1

漏洞概述

代码依旧位于jimureport-spring-boot-starter-1.5.6.jar的org.jeecg.modules.jmreport.desreport.a.a

由于jmreport/**无需鉴权，因此用户可在不登录的情况下通过积木报表这个组件提供的upload接口来进行文件上传

漏洞分析

来到文件上传接口，这段代码是一个文件上传接口，用于处理报表系统（JimuReport）中的文件上传请求。它支持多种存储方式（阿里云 OSS、MinIO、本地存储），并将上传后的文件访问路径返回给前端。

获取参数 “bizType”，如果为空，则从配置中获取 uploadType

我们这里是用的是本地上传的方法

文件经过处理后的url路径为jimureport/文件名+时间戳.txt​，而jimureport这个路径是需要身份认证才能访问的

有个点，通过upload上传的文件都会被放在opt/upFiles/jimureport这个文件夹下，如果我们能构造路径穿越将文件上传到一个允许未授权访问的目录，如jmreport，那么便可以实现未授权任意文件上传加任意文件访问
