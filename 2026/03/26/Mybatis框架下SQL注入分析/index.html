Mybatis框架下的SQL注入分析

MyBatis基础

什么是MyBatis

MyBatis 是一个优秀的持久层框架，它通过封装 JDBC将 SQL 语句与 Java 代码分离，通过 XML 或注解的方式配置 SQL，并自动完成输入参数映射和输出结果映射。

核心组件：

  • ​SqlSessionFactory：负责创建 SqlSession，内部持有 Configuration 和 Mapper 注册信息。
  • ​SqlSession：执行 SQL 的会话，提供了增删改查方法。
  • ​Mapper 接口：开发者定义的接口，MyBatis 通过动态代理生成实现类，将方法调用转换为 SQL 执行。
  • XML 映射文件：存放 SQL 语句、参数映射、结果映射等。

工作流程

构建SqlSessionFactory

MyBatis 从类路径加载 mybatis-config.xml​，读取全局配置，构建SqlSessionFactory对象

获取 SqlSession 和 Mapper 代理

通过sqlSessionFactory.openSession()​获取sqlSession​对象，再通过sqlSession​获取Mapper代理

执行 SQL

sql语句被写在了Mapper.xml里面，当执行sql语句时，会将方法调用转换为 SQL 执行

参数处理机制

MyBatis 提供了两种参数占位符：

  • ​ ​#{}​ ​：预编译占位符。MyBatis 会将其解析为 ?​，然后通过 PreparedStatement​ 设置参数，​自动对参数值进行转义，防止 SQL 注入。
  • ​ ​${}​ ​：字符串直接替换占位符。MyBatis 将 ${}​ 中的内容直接拼接到 SQL 语句中，​不经过预编译，因此存在 SQL 注入风险。

从这里可以看出，MyBatis会出现SQL注入的根本原因是因为开发者在某些场景下误用了 ​${}​ ​，或者未对动态拼接的内容进行严格校验

SQL查询过程

从方法调用处打个断点，我们在xml中编写的sql代码长这样select ${name} from mybatis.user

首先会走到MapperProxy，这里可以看到 Mapper 接口的方法被代理，根据方法名获取对应的 MapperMethod 并执行

根据 SqlCommandType​ 选择执行分支（SELECT / UPDATE / …），我这里是select，对于返回集合的方法，会进入 executeForMany

确定执行的方法以及返回的类型后会进入sqlSession层，这里通过 configuration.getMappedStatement(statement)​ 获取 MappedStatement，然后调用

executor.query(ms, wrapCollection(parameter), rowBounds, NO_RESULT_HANDLER)

进入org.apache.ibatis.executor.BaseExecutor​，这里会处理一级缓存，最终调用 doQuery​，doQuery​具体实现在子类如 SimpleExecutor 中。这里会创建

​StatementHandler​，并调用JDBC的Statement来执行sql查询

如果在xml中使用的是 ​#{}​ ​则会通过PreparedStatementHandler​调用PreparedStatement来执行查询

注意点

值得注意的是BoundSql里面会存放经过处理的sql语句，如果是由${}传入的参数会被直接拼接到sql语句中

而通过#{}​处理的传参，会显示 SQL 中包含 ?​，然后 DefaultParameterHandler​ 会设置参数，最终 PreparedStatement 执行，这极大的避免了sql注入的产生

其实到这里就可以看出端倪，如果使用了 ${}​，在构建 BoundSql​ 时参数就已经被拼接到 SQL 字符串中，后续即使使用 PreparedStatement​，也只是一个普通的字符串执行，失去了预编译保护。而使用 #{}​，StatementHandler​ 会创建 PreparedStatement​ 并通过 ParameterHandler​ 安全设置参数。因此，SQL 注入的风险在 ​MappedStatement​​ 生成 ​BoundSql​​ 的阶段就已经存在，selectList 只是传递了已经存在风险的 SQL。这跟JDBC导致的sql注入是不同的

导致sql注入的原因

依照上文所说，导致sql注入的罪魁祸首是${}​，那将这个都替换为#{}不就行了吗？

其实不然，mybatis防御sql注入的手段还是预编译那一套，而有些参数是没办法预编译的，具体原因可以看我之前写的笔记

order by 后为什么不能预编译（参数化）

我们来通过一段java代码来理解

ps.setString(1, username)会自动给值加上引号。比如假设username=“ls”，那么拼凑成的语句会是String sql = “ SELECT passwd FROM test_table1 WHERE username = ‘ls’ “;

再看order by，order by后一般是接字段名，而字段名是不能带引号的，比如 order by username；如果带上引号成了order by ‘username’，那username就是一个字符串不是字段名了，这就产生了语法错误。

所以order by后不能参数化的本质是：一方面预编译又只有自动加引号的setString()方法，没有不加引号的方法；而另一方面order by后接的字段名不能有引号。（至于为什么不弄个能不自动加引号的set方法那就不太懂了）

更本质的说法是：不只order by，凡是字符串但又不能加引号的位置都不能参数化；包括sql关键字、库名表名字段名函数名等等。

如何防御？将字段名设为白名单

一些可能存在注入的场景

动态表名 / 列名

业务需求需要根据用户选择动态查询不同表或字段

payload:user; DROP TABLE user; --

LIKE 模糊查询

搜索功能，用户输入关键词模糊匹配。

payload:%' OR '1'='1

ORDER BY / GROUP BY / HAVING 子句

前端支持按不同字段排序，用户通过请求参数传递排序字段和方向

payload:orderBy=id; DROP TABLE user;--

IN 子句

批量查询，用户传入多个 ID，如 ids=1,2,3。

payload:1) OR 1=1 --

注解中的字符串拼接

使用 @Select​、@Insert​ 等注解，但用 + 拼接用户输入。
