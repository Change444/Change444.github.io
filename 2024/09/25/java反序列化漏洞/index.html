初学java反序列化

漏洞原理

当一个类实现了Serializable接口，那么它将能够被序列化/反序列化，通过重写readObject方法，那么在反序列化这个类时将能执行恶意代码

为什么会产生安全问题?

只要服务端反序列化数据，客户端传递类的readObject中代码会自动执行，给予攻击者在服务器上运行代码的能力

可能的形式

1.入口类的readObject直接调用危险方法。

2.入口类参数中包含可控类，该类有危险方法，readObject时调用。

3.入口类参数中包含可控类，该类又调用其他有危险方法的类，readObiect时调用

疑虑

到这里便产生了个问题，在实际情况中，我们无法对对方java的类进行修改，我们该如何执行恶意语句呢?

答：利用反射

反射操作

一般情况下我们使用反射获取一个对象的步骤：

  • 获取类的 Class 对象实例

或

  • 根据 Class 对象实例获取 Constructor 对象（有参）
  • 或者直接通过Class对象实例类（空参）

而如果要调用某一个方法，则需要经过下面的步骤：

  • 获取方法的 Method 对象

若方法为私有方法

  • 利用 invoke 方法调用方法

获取属性或通过反射给属性赋值

  • 获取属性的Field对象

若属性为私有属性

  • 获取属性值
  • 修改属性值

我对于反射的理解

反射用于对类进行动态修改，相较于直接实例.属性名为属性赋值，反射能够做到对私有属性的访问与修改

构建链条（二）

虽然是rce了，但是其实还是有问题，因为checkSetValue方法是由我们反射获取的，而在真实情况中，我们没有权利大到控制服务器来反射反序列化对象的方法，所以还得围绕checkSetValue方法进行进一步探索。

还记得前面走到过TransformedMap的父类AbstractInputCheckedMapDecorator，这里面有个静态类MapEntry定义了setValue方法调用了checkSetValue。当时没跟仔细，后面看文章发现点击旁边的蓝色按钮还能进一步跟进，进去后发现setValue方法是重写它的父类AbstractMapEntryDecorator来的，然后再进一步，就能看到setValue来自Map的Entry接口

Entry其实就是键值对的意思·，当我们要修改某一个key的value值时，我们通常要先获取那个Entry然后调用Entry的setValue方法

AbstractInputCheckedMapDecorator中的MapEntry类重写了Entry的setValue方法，使得每次键值对调用SetValue时都会调用该对象的checkSetValue方法，而TransformedMap继承了AbstractInputCheckedMapDecorator，所以TransformedMap的对象也拥有这一特性，理论上来说，当调用获取TransformedMap实例的键值对并调用setValue方法时是可以触发rce的。

编写代码，然后在调用checkSetValue的地方下断点，可以看到程序进入的指定位置，并且成功rce，这样一样，如何调用checkSetValue的问题就解决了

要解决这个问题，我们现在可以这么想，如果有个类的transform方法可以无视输入的参数而返回Runtime.class，那么就可以无视setValue中的参数了

CC1–LazyMap

目标类还是InvokerTransformer，分歧点出现在transform方法的调用处，上面的用到的是TransformedMap类，而这里是LazyMap。

查找危险方法的用法，LazyMap类在get方法中调用了factory的transform方法，查找factory是哪来的

可以看到factory有构造方法传入，而构造方法的作用域是protected，所以要靠decorate函数来调用构造方法，这点跟之前有点像，我们只要采用差不多的方法就能构造LazyMap的实例

接下来寻找get方法的利用点，查找用法有点吓人哈，这边就不一个个找了，直接定位到AnnotationInvocationHandler类AnnotationInvocationHandler.invoke调用了get方法，而在前一条链子中我们知道memberValues是构造方法里传入的，所以我们现在只要考虑如何调用invoke就行

注意，AnnotationInvocationHandler实现了InvocationHandler，这个接口是用来动态代理的，当我们调用该类的代理类的任意方法时，都会被委派到AnnotationInvocationHandler的invoke方法

但是AnnotationInvocationHandler的实例被反序列化时不会自动执行invoke方法，还得找个类随便调用AnnotationInvocationHandler的一个方法

还是在这个类，readObject方法中就调用了任意类的entrySet方法。

但是下面会有点绕，我们需要在invoke方法里用到memberValues.get来进入LazyMap，所以memberValues应该是LazyMap的实例，但在这里，我们又要令memberValues成为AnnotationInvocationHandler的实例来调用invoke方法。听起来是不是有点矛盾？其实有解决办法，上面是不是提到了AnnotationInvocationHandler是代理类，我们可以让AnnotationInvocationHandler来代理LazyMap，这样问题就迎刃而解了
