Fastjson反序列化分析

简介

Fastjson 是阿里巴巴的开源 JSON 解析库，它可以解析 JSON 格式的字符串，支持将 Java Bean 序列化为 JSON 字符串，也可以从 JSON 字符串反序列化到 JavaBean，Fastjson不但性能好而且API非常简单易用，所以用户基数巨大，一旦爆出漏洞其影响对于使用了Fastjson的Web应用来说是毁灭性的。

Fastjson 使用

将类转为 json

在这里我们最常用的方法就是 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">JSON.toJSONString()</font>

将 json 反序列化为类

将 json 数据反序列化时常使用的方法为<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">parse()</font>、<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">parseObject()</font>、<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">parseArray()</font>

特性

  • 使用 <font style="background-color:rgb(247, 247, 247);">JSON.parse(jsonString)</font> 和 <font style="background-color:rgb(247, 247, 247);">JSON.parseObject(jsonString, Target.class)</font>，两者调用链一致，前者会在 jsonString 中解析字符串获取 <font style="background-color:rgb(247, 247, 247);">@type</font> 指定的类，后者则会直接使用参数中的class。
  • fastjson 在创建一个类实例时会通过反射调用类中符合条件的 getter/setter 方法，其中 getter 方法需满足条件：方法名长于 4、不是静态方法、以 <font style="background-color:rgb(247, 247, 247);">get</font> 开头且第4位是大写字母、方法不能有参数传入、继承自 <font style="background-color:rgb(247, 247, 247);">Collection|Map|AtomicBoolean|AtomicInteger|AtomicLong</font>、此属性没有 setter 方法；setter 方法需满足条件：方法名长于 4，以 <font style="background-color:rgb(247, 247, 247);">set</font> 开头且第4位是大写字母、非静态方法、返回类型为 void 或当前类、参数个数为 1 个。具体逻辑在 <font style="background-color:rgb(247, 247, 247);">com.alibaba.fastjson.util.JavaBeanInfo.build()</font> 中。
  • 使用 <font style="background-color:rgb(247, 247, 247);">JSON.parseObject(jsonString)</font> 将会返回 JSONObject 对象，且类中的所有 getter 与setter 都被调用。
  • 如果目标类中私有变量没有 setter 方法，但是在反序列化时仍想给这个变量赋值，则需要使用 <font style="background-color:rgb(247, 247, 247);">Feature.SupportNonPublicField</font> 参数。
  • fastjson 在为类属性寻找 get/set 方法时，调用函数 <font style="background-color:rgb(247, 247, 247);">com.alibaba.fastjson.parser.deserializer.JavaBeanDeserializer#smartMatch()</font> 方法，会忽略 <font style="background-color:rgb(247, 247, 247);">_|-</font> 字符串，也就是说哪怕你的字段名叫 <font style="background-color:rgb(247, 247, 247);">_a_g_e_</font>，getter 方法为 <font style="background-color:rgb(247, 247, 247);">getAge()</font>，fastjson 也可以找得到，在 1.2.36 版本及后续版本还可以支持同时使用 <font style="background-color:rgb(247, 247, 247);">_</font> 和 <font style="background-color:rgb(247, 247, 247);">-</font> 进行组合混淆。
  • fastjson 在反序列化时，如果 Field 类型为 <font style="background-color:rgb(247, 247, 247);">byte[]</font>，将会调用<font style="background-color:rgb(247, 247, 247);">com.alibaba.fastjson.parser.JSONScanner#bytesValue</font> 进行 base64 解码，对应的，在序列化时也会进行 base64 编码。

@type的作用

<font style="color:rgb(21, 167, 167);">@type</font>是<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">fastjson</font>中的一个特殊注解，用于标识<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">JSON</font>字符串中的某个属性是一个<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">Java</font>对象的类型。具体来说，当<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">fastjson</font>从<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">JSON</font>字符串反序列化为<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">Java</font>对象时，如果<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">JSON</font>字符串中包含<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">@type</font>属性，<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">fastjson</font>会根据该属性的值来确定反序列化后的<font style="color:rgb(21, 167, 167);background-color:rgb(247, 248, 250);">Java</font>对象的类型。

漏洞利用

参考Fastjson 反序列化漏洞 · 攻击Java Web应用-[Java Web安全]

如何才能触发反序列化得到的恶意类中的恶意函数呢？

Fastjson 反序列化漏洞的核心在于其 autoType 功能。当 autoType 开启时，Fastjson 会根据 JSON 数据中的 @type 字段来实例化对应的 Java 对象。

由前面知道，在某些情况下进行反序列化时会将反序列化得到的类的构造函数、getter方法、setter方法执行一遍，如果这三种方法中存在危险操作，则可能导致反序列化漏洞的存在。

我在学生类里添加一个恶意getter，能被正常调用

关于Poc怎么写的问题

我们的 PoC 是把恶意代码放到一个 json 格式的字符串里面，开头是要接 @type 的。这里其实赋值就代表我们不需要通过反射来修改值，而是可以直接赋值。

@type 之后是我们要去进行反序列化的类，会获取它的构造函数、getter 与 setter 方法。

各版本漏洞分析

Fastjson-1.2.24

TemplatesImpl 反序列化

在CB1我们讲过TemplatesImpl存在一个getter能够调用TemplatesImpl.newTransformer达到加载恶意字节码的目的，这个方法就是TemplatesImpl.getOutputProperties,这个getter在被反序列化时能够被自动调用，又因为能通过修改json字符串而对TemplatesImpl中的变量赋值（如果是私有变量且没有setter方法，还需添加Feature.SupportNonPublicField标识），从而绕过方法中的判断，使程序走到实例化恶意字节码。

回顾一下调用链<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">getOutputProperties()</font> -> <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">newTransformer()</font> -> <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">getTransletInstance()</font> -> <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">defineTransletClasses()</font> / <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">EvilClass.newInstance()</font>.

其中，为了满足漏洞点触发之前不报异常及退出，我们还需要满足 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">_name</font> 不为 null ，<font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">_tfactory</font> 不为 null 。这个可以在cc3里面找到为什么，这里就不重复了

由于部分需要我们更改的私有变量没有 setter 方法，需要使用 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">Feature.SupportNonPublicField</font> 参数。

JdbcRowSetImpl 反序列化

这个涉及到了JNDI，在JdbcRowSetImpl这个类有个符合条件的setter调用了InitialContext.lookup，由于dataSourceName可以被json字符串指定，发现这个漏洞时我们可以自己起一个rmi或者ldap服务，利用Reference类型进行攻击

小结

TemplatesImpl 是有一点限制的，需要对方的代码里面能够让我们加载的私有的 getter/setter。也就是需要这个参数 Feature.SupportNonPublicField

第二种攻击方式，需要针对 jdk 版本。

Fastjson-1.2.25<= 1.2.41

在版本 1.2.25 中，官方对之前的反序列化漏洞进行了修复，引入了 checkAutoType 安全机制，默认情况下 autoTypeSupport 关闭，不能直接反序列化任意类，而打开 AutoType 之后，是基于内置黑名单来实现安全的，fastjson 也提供了添加黑名单的接口。

看一下 <font style="background-color:rgb(247, 247, 247);">checkAutoType()</font> 的逻辑，如果开启了 autoType，先判断类名是否在白名单中，如果在，就使用 <font style="background-color:rgb(247, 247, 247);">TypeUtils.loadClass</font> 加载，然后使用黑名单判断类名的开头，如果匹配就抛出异常。如果既不在黑名单也不在白名单那将继续后续的流程

如果没开启 autoType ，则是先使用黑名单匹配，再使用白名单匹配和加载。最后，如果要反序列化的类和黑白名单都未匹配时，只有开启了 autoType 或者 expectClass 不为空也就是指定了 Class 对象时才会调用 <font style="background-color:rgb(247, 247, 247);">TypeUtils.loadClass</font> 加载。

接着跟一下 <font style="background-color:rgb(247, 247, 247);">loadClass</font> ，这个类在加载目标类之前为了兼容带有描述符的类名，使用了递归调用来处理描述符中的 <font style="background-color:rgb(247, 247, 247);">[</font>、<font style="background-color:rgb(247, 247, 247);">L</font>、<font style="background-color:rgb(247, 247, 247);">;</font> 字符。

因此就在这个位置出现了逻辑漏洞，攻击者可以使用带有描述符的类绕过黑名单的限制，而在类加载过程中，描述符还会被处理掉。因此，漏洞利用的思路就出来了：需要开启 autoType，使用以上字符来进行黑名单的绕过。

最终的 payload 其实就是在之前的 payload 类名上前后加上<font style="background-color:rgb(247, 247, 247);">L</font>和<font style="background-color:rgb(247, 247, 247);">;</font>即可：

fastjson-1.2.42

在 checkAutoType 中加入判断，如果类的第一个字符是 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">L</font> 结尾是 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">;</font>，则使用 substring进行了去除。

双写绕过即可

fastjson-1.2.47

在 fastjson 不断迭代到 1.2.47 时，爆出了最为严重的漏洞，可以在不开启 AutoTypeSupport 的情况下进行反序列化的利用。

问题出现在<font style="color:rgb(51, 51, 51);">checkAutoType</font>方法，我们分析一下该方法中的黑白名单检测

当autoTypeSupport为true时，当反序列化的类在黑名单中，且 TypeUtils.mappings 中没有该类的缓存时，才会抛出异常。如果TypeUtils.mappings缓存了该类，将会在 TypeUtils.mappings和deserializers查找该类，若找到就会 return。如果我们能将恶意类写入TypeUtils.mappings，那将能绕过autoTypeSupport为true时的判断

而当autoTypeSupport为false时，当反序列化的类在黑名单中则会直接抛出异常，但由于程序是顺序执行的，会先在TypeUtils.mappings和deserializers查询和return类，于是这个检测也被绕过了

但要怎么将恶意类写入到TypeUtils.mappings中呢，我们可以用一下这个payload

在调用DefaultJSONParser.parserObject()函数时，其会对JSON数据进行循环遍历扫描解析，第一次会扫描到java.lang.Class这个类，因为这个类不在黑名单中，所以能很轻松的通过检测，并赋值给clazz

然后跳出checkAutoType，接着会获取clazz的deserializer，deserializer的值是MiscCodec，这个类是用来处理一些乱七八糟类的反序列化类，其中就包括 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">Class.class</font> 类，然后就能走到MiscCodec.deserialze中

这个类会读取val的值，并且如果clazz == Class.class将会调用 <font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">loadClass</font> 方法，将 strVal 进行类加载并缓存

在<font style="color:rgb(51, 51, 51);">TypeUtils.</font><font style="color:rgb(51, 51, 51);background-color:rgb(247, 247, 247);">loadClass</font>中strVal的值会被放到TypeUtils.mappings中，从而使得第二次遍历json数据时能够绕过黑名单检查直接加载<font style="color:rgb(51, 51, 51);">com.sun.rowset.JdbcRowSetImpl</font>

由于1.2.47这个洞能够在不开启AutoTypeSupport实现RCE，因此危害十分巨大，看看是怎样修的。1.2.48中的修复措施是，在loadClass()时，将缓存开关默认置为False，所以默认是不能通过Class加载进缓存了。同时将Class类加入到了黑名单中。
